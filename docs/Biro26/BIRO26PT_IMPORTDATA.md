# `BIRO26PT_importData` — универсальный импорт файлов неизвестной структуры

## 0. Назначение

Пакет решает задачу: **импортировать группу файлов, точная структура которых
заранее не известна**, при условии что в файле есть хотя бы один из сигналов:

1. **понятные имена колонок** в шапке (RO/RU/EN синонимы);
2. **известный порядок колонок** (позиционный layout);
3. **хотя бы один узнаваемый товар** в данных (совпадение по артикулу
   `CODVECHI` или по названию `DENUMIREA`).

Результат — три действия (по наличию данных): **новые позиции**, **обновление
цен** по существующим, **добавление штрихкодов**.

Все сообщения/комментарии кода — **RO + EN** (правило проекта). Данные (названия
товаров) — как в источнике.

---

## 1. Архитектура — 2 слоя

PL/SQL не читает xlsx, поэтому ответственность разделена:

| Слой | Объект | Задача |
|---|---|---|
| **Loader (Python)** | `biro26pt_loader.py` | Грузит любой xlsx/csv в «сырой» staging **без интерпретации** |
| **Detector+Importer (PL/SQL)** | `BIRO26PT_importData` | Детекция колонок + импорт (вся логика в БД) |

### Сырой staging (generic)
- `BIRO26PT_RAW(load_id, src_file, sheet, row_no, c0..c15)` — все ячейки как текст.
- `BIRO26PT_HEADER(load_id, src_file, col_idx, header_text)` — строка заголовков.
- `BIRO26PT_FILE(load_id, src_file, sheet, n_rows, n_cols, loaded_at)` — реестр загрузок.

Один загрузчик обслуживает все будущие батчи; `load_id` уникален на «файл+лист».

### Служебные таблицы
- `BIRO26PT_COLMAP(pattern, logical_field, prio)` — словарь синонимов заголовков.
- `BIRO26PT_LAYOUT(sig_name, col_idx, logical_field)` — известные позиционные раскладки.
- `BIRO26PT_MAP(load_id, logical_field, col_idx, strategy, confidence)` — **результат детекции**.
- `BIRO26PT_STG(...)` — проекция RAW в «goods»-форму (для переиспользования `YBIRO_Import_Marfa`).
- `BIRO26PT_LOG(...)` — журнал детекции/импорта (аудит).

## 2. Детекция колонок — 3 стратегии (`detect_columns`)

Результат: маппинг `логическое поле → cNN` в `BIRO26PT_MAP`. Поля:
`ARTICOL, DENUMIRE, BARCODE, ANGRO, ONLINE, RETAIL, VAT, URL` (+ `IGNORE` игнорируется).
`URL` — ссылка на изображение товара: детектируется по шапке (`url|image|link`)
или по содержимому (колонка, где ≥70% значений начинаются с `http(s)://`).

**Стратегия 1 — по имени колонки** (высший приоритет).
`LOWER(header) LIKE pattern` по `BIRO26PT_COLMAP`; меньший `prio` выигрывает.
Двойная редукция: каждая колонка берёт одно лучшее поле, затем каждое поле — одну
лучшую колонку. Пример синонимов: `артикул|articol→ARTICOL`,
`название|denumire→DENUMIRE`, `закупки без ндс|angro→ANGRO`, `priceonline|online→ONLINE`,
`розничная|retail|raft→RETAIL`, `штрих|barcode|cod de bare→BARCODE`.

**Стратегия 3 — по содержимому / якорю-товару** (fallback, если после шапки нет
`ARTICOL` или `DENUMIRE`). На выборке (`g_sample_rows`) для каждой ещё свободной
колонки считаются доли совпадений:
- с `TMS_UNIVERS.CODVECHI` → кандидат `ARTICOL`;
- с `TMS_UNIVERS.DENUMIREA` (норм.) → `DENUMIRE` (**якорь «известный товар»**, ≥1 хит);
- regex `^\d{8,14}$` → `BARCODE`;
- числовые деньги → цены; три ценовые колонки упорядочиваются по медиане:
  `angro < online ≤ retail`.

**Стратегия 2 — по известному порядку** (последний резерв, если `ARTICOL` и
`DENUMIRE` так и не найдены). Берётся раскладка из `BIRO26PT_LAYOUT`, чьё число
колонок укладывается в файл (напр. историч. формат ШК `[1]=BARCODE,[2]=ARTICOL,[3]=DENUMIRE`).

Приоритет: **шапка → содержимое → layout**; недостающие поля добираются по порядку.
Каждое решение пишется в `BIRO26PT_LOG` (стратегия + confidence) — dry-run показывает,
как понят каждый файл. Если не найдены ни артикул, ни название — файл помечается
«не распознан» и пропускается.

## 3. Пайплайн импорта (`import_file` / `import_folder`)

1. **`build_stg`** — динамическая проекция `RAW → BIRO26PT_STG` по маппингу
   (числовые цены парсятся, `retail1` остаётся текстом — `YBIRO_Import_Marfa.parse_price`
   обрабатывает формат вида `1.013.00`). Берутся строки, где есть артикул ИЛИ название.
2. **`classify`** — статус строки по `CODVECHI=articol`:
   `NEW` (нет в справочнике) / `EXISTING` (ровно 1 код — привязывается) /
   `AMBIGUOUS` (>1 код — пропуск) / `NOARTICOL`. Плюс подсчёт «цена изменилась»
   (файл `retail` vs текущая `PRETV` последнего периода).
3. **Запись в прод** (`do_writes`, только при `p_commit=TRUE`):
   - новые позиции: `COD` из `ID_TMS_UNIVERS` → `import_univers` + `import_mpt`;
   - цены: **автоматическое обновление** — новым товарам первая цена; существующим
     **новый период `datastart=дата вступления`** (по умолч. сегодня, или из `p_date`)
     **всегда, когда розничная цена в файле отличается от текущей** (выше ИЛИ ниже);
     повторный импорт в тот же день **обновляет значения периода этого дня** (MERGE);
     после вставки **предыдущий период закрывается**
     (`DATAEND = новый_старт − 1` по правилу `LEAD(datastart) − 1` per `(codprice, sc)`);
     если артикул встречается в файле несколько раз с разными ценами — берётся максимум;
   - **изображения из `URL` → `TMS_MPT_TVR.IE_LINKADRES`** (картинка карточки товара,
     видна в магазине/backoffice): новым товарам — всегда; существующим — только если
     изображения ещё нет (ручные картинки не перезаписываются);
   - **маркер «новые товары» = `TMS_MPT.MATGR1 = 1`** (виден через `VMS_MPT`). Это и есть
     **виртуальный «PRODUSE NOI»** — узла дерева нет, «новинки» — только признак `MATGR1=1`,
     по которому работают фильтры «🆕 Produse noi» в магазине и backoffice.
     `p_mark_all_new=TRUE` → помечаются все строки файла; `FALSE` → только новые позиции;
   - **размещение в дереве — в СВОИ реальные узлы по `GRUPA`**: `ensure_group(имя)` ищет
     существующий узел с таким именем на любом уровне дерева (при нескольких одноимённых
     выбирает самый наполненный, затем меньший `id1`) и лишь при полном отсутствии имени
     создаёт узел верхнего уровня. Физический узел «PRODUSE NOI» **не создаётся**;
   - **генерация штрихкода**: для новых позиций без ШК генерируется валидный **EAN-13**
     (префикс `20` — внутренний in-store диапазон, + последовательность из
     `BIRO26PT_EAN_SEQ` + контрольная цифра); при наличии колонки `BARCODE` в файле —
     дополнительно `import_barcodes` (идемпотентно, защита `TMS_BARCODE_UNIQ`).

Переиспользуется проверенный пакет `YBIRO_Import_Marfa` (его `g_tbl_goods` и
`g_col_*` временно наводятся на `BIRO26PT_STG`).

## 4. Безопасность / идемпотентность

- **Dry-run по умолчанию** (`p_commit=FALSE`): детекция + классификация + отчёт, в
  продакшен **ничего не пишется**.
- Все вставки под `NOT EXISTS`; **удалений нет**; цены — периодами; штрихкоды аддитивны.
- Детекция и импорт журналируются в `BIRO26PT_LOG`.
- Откат цен: `YBIRO_Import_Marfa.rollback_pricelist(codprice)`.

## 5. Использование

```bash
# 1) загрузка файлов (shell)
python biro26pt_loader.py /Users/pt/Projects.AI/BIRO26/Set_data_import/3
```
```sql
SET SERVEROUTPUT ON
-- 2) dry-run (анализ, без записи):
BEGIN BIRO26PT_importData.import_folder(p_commit => FALSE); END;
/
-- 3) реальный импорт одного файла:
--    p_mark_all_new: TRUE = все строки файла -> MATGR1=1 (по умолч.), FALSE = только новые позиции.
--    p_date: дата вступления цены; NULL = дата загрузки (сегодня).
BEGIN BIRO26PT_importData.import_file(
        p_load_id => 4, p_grupa => 'Hartie colorata', p_codprice => 1,
        p_commit => TRUE, p_mark_all_new => TRUE, p_date => NULL); END;
/
-- Фильтр «produse noi» для интерфейса:
SELECT * FROM vms_mpt WHERE matgr1 = 1;
-- MD-описание алгоритма из тела пакета:
SELECT BIRO26PT_importData.algo_md FROM dual;
```

Функция `algo_md RETURN CLOB` возвращает это описание (RO+EN) прямо из БД.

| Перекрытий периодов | 0 |

> **Правка после первого прогона:** изначально 523 товара были привязаны только к
> физическому узлу «PRODUSE NOI» и не состояли ни в одном настоящем узле дерева.
> Исправлено: пакет теперь кладёт новинки в свои реальные узлы по `GRUPA`
> (`ensure_group`), а «PRODUSE NOI» стал виртуальным (только `MATGR1=1`). Данные
> batch-4 отремонтированы (delete+insert одной транзакцией, т.к. нативный триггер
> `TR_TMS_SYSGRP_B` запрещает менять ключевые поля и сам считает `ID1/ID2`); пустой
> физический узел «PRODUSE NOI» удалён. Выбор узла (277,64) из трёх одноимённых
> обоснован маппингом фида `BIRO26_GOODS` (304 из 379 позиций «Hartie colorata» — там).

Пример EAN: `cod 288657 → 2000000004068`. Пример keep-higher: `cod 165690` файл 42.41 <
текущая 42.4137 → **период не создан**, сохранена текущая.

## 7. Важный нюанс окружения: charset БД

БД `OFFICEPLUS` имеет `NLS_CHARACTERSET = CL8MSWIN1251` (однобайтовая кириллица),
**не UTF-8**. Поэтому:

- Кириллицу в справочники (`BIRO26PT_COLMAP` и т.п.) надёжно грузить через
  **python-oracledb** (сам конвертирует Unicode→win1251). Вставка кириллицы через
  SQLcl-heredoc **портит байты**, и `LIKE` перестаёт матчить.
- `biro26pt_loader.py` использует этот путь, поэтому заголовки/данные ложатся корректно.
