# Техническое задание

## Трёхъязычное приложение управления номенклатурой, прайс-листами и импортом (БД OfficePlus / UNA.md)

**Версия:** 1.0 · **Дата:** 2026-06-28 · **СУБД:** Oracle 23ai (`OFFICEPLUS@//orange.una.md:4024/cloudbd.world`)

> Языки интерфейса и данных: **RU / RO / EN**.
> Все сообщения и комментарии в коде БД — только **RO + EN** (см. `AGENTS.md` / `CLAUDE.md`).
> Сопутствующие документы: `IMPORT_TMS_UNIVERS.md` (методика импорта), `YBIRO_Import_Marfa.pkg.sql` (пакет).

---

## 1. Назначение и цели

Приложение автоматизирует загрузку товаров из внешнего фида (стейджинг-таблица `BIRO26_GOODS`)
в корневой справочник ERP-системы UNA.md (`TMS_UNIVERS`) и сопутствующее построение
прайс-листов, с дальнейшим расширением до ввода приходов и продаж в стиле 1С.

**Цели:**
1. Управляемый, идемпотентный, повторяемый импорт номенклатуры и цен.
2. Единый стабильный ключ связи «фид ↔ справочник» (`COD_UNIVERS`).
3. Настраиваемый маппинг полей без перекомпиляции.
4. Полное логирование операций и их просмотр.
5. Трёхъязычный интерфейс (RU/RO/EN) поверх двуязычных данных (`DENUMIREA`=RO, `NAMERUS`=RU).
6. Перспектива: ввод приходов/продаж (документы `TMDB_*`).

## 4. Настраиваемый маппинг полей

Маппинг и константы **не хардкодятся**, а конфигурируются. На уровне БД это уже
реализовано как публичные переменные пакета (см. §7); приложение даёт UI поверх них.

### 4.1 Маппинг «источник → справочник `TMS_UNIVERS`»
| Поле `TMS_UNIVERS` | Источник / правило | Перем. пакета |
|---|---|---|
| `COD` | `BIRO26_GOODS.COD_UNIVERS` (seq `ID_TMS_UNIVERS`) | `g_col_key`, `g_seq_key` |
| `CODVECHI` | `ARTICOL`, обрезка до 20 | `g_col_articol`, `g_len_codvechi` |
| `DENUMIREA` | `DENUMIRE`, обрезка до 160 | `g_col_denumire`, `g_len_denumire` |
| `UM` | константа `'buc.'` | `g_um` |
| `GR1` | константа `'TVR'` | `g_gr1` |
| `TIP` | константа `'P'` | `g_tip` |
| `CACCESS` | константа `'11100'` (NOT NULL) | `g_caccess` |
| `CODTVA` | константа `'A'` | `g_codtva` |
| `NAMERUS, GR2, ISARHIV, …` | NULL | — |

### 4.2 Маппинг «источник → прайс `VTPR1D_PERPRLIST`»
| Колонка | Источник | Перем. пакета |
|---|---|---|
| `PRETV` | `RETAIL1` (парсинг, формат `1.013.00`) | `g_col_retail` |
| `PRETV1` | `ANGRO` | `g_col_angro` |
| `PRETV2` | `IONLINE` | `g_col_ionline` |
| `CODGRP` | группа по `BRAND` | `g_col_brand` |
| `SC` | `COD_UNIVERS` | `g_col_key` |
| `DATASTART/DATAEND` | период | `g_date_start/g_date_end` |

### 4.3 Требования к UI маппинга
- Таблица соответствий: «целевое поле ← источник/константа», с типом и правилом (обрезка/парсинг).
- Чтение/запись значений переменных пакета (`g_*`) как профиля маппинга.
- Сохранение нескольких именованных профилей маппинга (на разные фиды).
- Валидация: длины, NOT NULL, конвертируемость числовых полей.

## 6. Функциональные требования — интерфейсы

### 6.1 Интерфейс работы с источником (`BIRO26_GOODS`)
- Просмотр/поиск/фильтр строк фида (грид: `ID, ARTICOL, DENUMIRE, BRAND, FURNIZOR, ANGRO, IONLINE, RETAIL1, STOC, COD_UNIVERS`).
- Импорт нового фида (CSV/XLSX) в стейджинг.
- **Предпроверка** данных: вызов `YBIRO_Import_Marfa.validate_input` — отчёт:
  пустые имена, длинные `ARTICOL/DENUMIRE`, пустой бренд, отсутствие ключа, непарсируемые цены.
- **Нормализация:** `prepare_input` (пустой бренд → маркер, `TRIM`).
- **Назначение ключей:** `assign_keys` (заполнение `COD_UNIVERS` из секвенции).
- Индикация статуса строки: «новая / уже в справочнике / конфликт артикула».

### 6.2 Интерфейс справочника (`TMS_UNIVERS` + `TMS_MPT`)
- Грид товаров `TIP='P'` с фильтрами по `GR1`, бренду, архиву (`ISARHIV`).
- Карточка товара: поля `TMS_UNIVERS` + связанная `TMS_MPT` (цены, счета, ШК, единицы).
- Двуязычные названия (`DENUMIREA`/`NAMERUS`).
- **Запуск импорта** товаров: `import_univers` (идемпотентно по `COD = COD_UNIVERS`).
- **Удаление заблокировано** триггерами БД → вместо удаления **архивирование**:
  `archive_univers` (значение `ISARHIV`; `'2'` блокируется триггером).
- Сервис очистки названий: `fix_denumirea_confusables` — замена кириллических
  «обманок» (`С→C`) на латиницу только в строках с ≤ N кириллических символов
  (русские названия не трогаются).

### 6.3 Интерфейс групп (`VPR01M_GROUPS`, дерево `TMS_SYSGRP*`)
- Просмотр/редактирование групп прайса (`CODPRICE, CODGRP, GRPNAME`).
- Построение групп из брендов фида: `import_groups` (нумерация `CODGRP` продолжается с максимума).
- Слияние групп (напр. устранение опечаток брендов: `Biroux-MT → Birolux-MT`) с
  переносом цен и удалением пустой группы.
- Просмотр дерева категорий `TMS_SYSGRP/TMS_SYSGRPH` (иерархия 5 уровней) — только чтение в фазе 1.

### 6.4 Интерфейс поставщиков (`TMS_ORG`, `FURNIZOR`)
- Справочник поставщиков из `TMS_ORG` (реквизиты, банки, счета `TMS_ORG_ACCOUNTS`).
- Сопоставление `BIRO26_GOODS.FURNIZOR` ↔ `TMS_ORG` (маппинг имён поставщиков на коды организаций).
- Привязка товара к поставщику; цены поставщика (`TMS_ORG_PRICES`).
- Отчёты: товары по поставщику, поставщики по группе.

### 6.5 Отдельный интерфейс прайс-листов (`VPR01M_GROUPS → VPR1D_PRDATE → VTPR1D_PERPRLIST`)
Самостоятельный модуль, цепочка из трёх шагов:
1. **Группы** прайса (`VPR01M_GROUPS`) — `import_groups`.
2. **Период** (`VPR1D_PRDATE`) — `import_dates` (дата начала, открытый конец `01.01.3000`).
3. **Цены** (`VTPR1D_PERPRLIST`) — `import_prices`:
   - одна цена на товар (`SC`), дедуп по `SC` (при дубле артикула — макс. розница);
   - запись через `INSTEAD OF` триггер представления;
   - `PRETV/PRETV1/PRETV2` ← `RETAIL1/ANGRO/IONLINE`.
- Грид цен с фильтром по `CODPRICE/CODGRP/период`, редактирование `PRETV*`.
- **Откат прайса:** `rollback_pricelist(p_codprice)` (цены → даты → группы; полностью обратимо).
- Учитывать PK `(CODPRICE, CODGRP, SC, DATASTART)` и отсутствие уникальности `CODVECHI` для товаров.

### 6.6 Единый запуск
- Кнопка «Полный импорт»: `import_all(p_codprice, p_date_start, p_date_end, p_prepare)`
  — выполняет всю цепочку §6.1→§6.5 с логированием каждого шага.

## 7. Интеграция с пакетом `YBIRO_Import_Marfa`

Вся бизнес-логика — в пакете; приложение его **вызывает**, а не дублирует SQL.

### 7.1 Настраиваемые переменные шапки (профиль конфигурации)
- **Идентификаторы:** `g_tbl_goods`, `g_col_key`, `g_col_id`, `g_col_brand`,
  `g_col_articol`, `g_col_denumire`, `g_col_angro`, `g_col_ionline`, `g_col_retail`, `g_seq_key`.
- **Значения/константы:** `g_codprice`, `g_um`, `g_gr1`, `g_tip`, `g_caccess`, `g_codtva`,
  `g_date_start`, `g_date_end`, `g_group_type`, `g_empty_brand`, `g_len_codvechi`,
  `g_len_denumire`, `g_isarhiv_arc`, `g_isarhiv_lock`, `g_confus_max_cyr`.
- Процедуры, работающие с входной таблицей, используют динамический SQL из этих имён.

### 7.2 Публичные процедуры/функции
`parse_price`, `fix_confusables`, `prepare_input`, `validate_input`, `assign_keys`,
`import_univers`, `import_groups`, `import_dates`, `import_prices`, `import_all`,
`rollback_pricelist`, `archive_univers`, `fix_denumirea_confusables`.

### 7.3 Требование на доработку пакета
Перевести вывод с `DBMS_OUTPUT` на запись в таблицу логов (§8) через
**автономную транзакцию**, чтобы операции протоколировались независимо от commit/rollback
основной транзакции, а UI мог читать прогресс в реальном времени.

## 8. Логирование

### 8.1 Таблица логов (создать)
```sql
CREATE TABLE YBIRO_IMPORT_LOG (
  id          NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  ts          TIMESTAMP DEFAULT SYSTIMESTAMP,   -- RO: data/ora / EN: timestamp
  run_id      NUMBER,                            -- RO: id rulare / EN: run id
  step        VARCHAR2(40),                      -- RO: pas / EN: step (import_univers, ...)
  level_code  VARCHAR2(10),                      -- INFO / WARN / ERROR
  rows_cnt    NUMBER,                            -- RO: randuri afectate / EN: affected rows
  message_ro  VARCHAR2(400),
  message_en  VARCHAR2(400),
  db_user     VARCHAR2(60) DEFAULT USER,
  err_code    NUMBER,
  err_text    VARCHAR2(2000)
);
```
- `run_id` — из секвенции; один прогон `import_all` = один `run_id`.
- Сообщения — RO + EN (правило проекта).
- Запись через процедуру `log(...)` с `PRAGMA AUTONOMOUS_TRANSACTION`.

### 8.2 Требования к логированию
- Каждый шаг пакета пишет старт/итог/ошибку с числом затронутых строк.
- Ошибки (`SQLCODE/SQLERRM`) фиксируются с `ROLLBACK` шага, прогон помечается `ERROR`.
- Долгие операции (импорт цен ~78 000 строк) — прогресс по этапам.

## 9. Просмотр логов (интерфейс)

- Грид прогонов: `run_id`, время, пользователь, статус (по max `level_code`), длительность.
- Детализация прогона: шаги, уровни, число строк, сообщения (RU/RO/EN — RU как перевод EN на UI).
- Фильтры: по дате, статусу, шагу, пользователю; полнотекстовый поиск по сообщению.
- Подсветка `WARN`/`ERROR`, экспорт в CSV.
- Просмотр в реальном времени (автообновление) во время выполнения импорта.

## 10. Перспектива: ввод приходов и продаж (стиль 1С)

Фаза 2. Документный учёт поверх существующих таблиц `TMDB_*`.

### 10.1 Объекты
- `TMDB_DOCS` — шапки документов (тип, номер, дата, контрагент).
- `TMDB_CM` — строки/проводки (субконто `SC` = `TMS_UNIVERS.COD`, количество, сумма).
- `TMDB01M_VINZ` — продажи.
- Секвенции `ID_TMDB_DOCS`, `ID_TMDB_CM`.

### 10.2 Функционал
- **Приход (поступление):** документ с поставщиком (`TMS_ORG`), строки товаров
  (`SC`), количество, цена закупки, склад; обновление остатков.
- **Продажа/реализация:** документ с ценой из активного прайс-листа
  (`VTPR1D_PERPRLIST`), списание остатков.
- Журнал документов (как в 1С): список, фильтры, проведение/распроведение, печать.
- Контроль остатков и цен на дату документа.
- Все операции — через выделенный пакет (`YBIRO_DOC_*`), по аналогии с импортом,
  с логированием (§8) и соблюдением триггеров `TMS_UNIVERS`/`TMDB_*`.

### 10.3 Требование
До разработки фазы 2 — отдельное обследование структуры `TMDB_*` и правил
проводок (триггеры, контроль субконто), т.к. это ядро учёта ERP.

## 11. Нефункциональные требования

- **Производительность:** массовые операции — set-based SQL/пакет (импорт цен ~78k строк ≈ 8 c).
- **Идемпотентность:** повторный запуск любого шага не создаёт дублей (`NOT EXISTS`).
- **Целостность:** не обходить триггеры БД (запрет удаления товаров — только архив).
- **Безопасность:** параметризованные запросы/bind-переменные; пароль БД — в защищённом
  хранилище (Keychain/secret store), не в коде.
- **Трёхъязычность:** полное покрытие RU/RO/EN UI; код БД — RO + EN.
- **Конфигурируемость:** маппинг и константы — через профили (переменные пакета), без перекомпиляции.
- **Аудит:** все изменения справочника/прайса логируются (кто, когда, что, сколько строк).
