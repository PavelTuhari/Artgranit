10. Расширения: изображения, мастер импорта, AI-источники

Просмотр изображений

Мастер импорта (wisepim-стиль)

Вкладка «Import (asistent)» — 4 шага: Sursă → Mapare (таблица «целевое поле ← колонка источника» с авто-подбором и превью примеров) → Verificare (validate_input) → Import (import_univers). Маппинг сохраняется в активный профиль YBIRO_MAP_*.

Любой SELECT как источник (+ AI)

В шаге «Sursă» — блок «Sursă nouă (SELECT)»: вставка read-only SELECT (guard: один оператор, запрет DML/DDL), AI-черновик описания (.md в docs/Biro26/sources/), AI-подбор маппинга колонок. SELECT материализуется во VIEW V_BIRO26_SRC_<name>, определение хранится в YBIRO_SRC_DEF (deploy: deploy_biro26_sources.py, требует привилегию CREATE VIEW). AI — через ai_helper (Selenium) с эвристическим fallback по именам колонок, поэтому мастер работает и без AI.

Новые эндпоинты: GET /api/biro26/source/columns|sample, GET/POST /api/biro26/sources, POST /api/biro26/sources/sample|ai-draft-md|ai-suggest-mapping.

Прайс-лист в стиле Windows + импорт картинок

Вкладка «Listă de prețuri» — мастер-деталь как в нативном приложении: слева прайс-листы (VPR0M_PRICES) и группы, справа строки цен с наименованием товара и миниатюрой; клик по строке — карточка товара с изображением. Эндпоинт GET /api/biro26/prices/lists.

Импорт ссылок на картинки: ссылки из фида (BIRO26_GOODS.PHOTO_URL, иначе IMAGE_LINK) импортируются в TMS_MPT_TVR.IE_LINKADRES (база вью VMS_MPT_TVR; колонка расширена до VARCHAR2(1000)) — set-based MERGE по COD = COD_UNIVERS, идемпотентно. Эндпоинт POST /api/biro26/images/import; в мастере импорта запускается автоматически после «Import nomenclator», либо вручную кнопкой «Импортировать картинки». Так ссылки на изображения подтягиваются при каждом импорте.

Товар / Остатки (UN$SOLD.GET_SOLDT)

Две вкладки: «Marfă / Stoc» — грид товаров в стиле исходного Excel-экспорта (фото, артикул, наименование, тамож./товарная группа, ед.изм., кол-во, закупка без/с НДС, онлайн-цена, розница, производитель, ставка НДС) и «Stoc (calcul)» — панель расчёта остатков.

Важная особенность Oracle: формула остатков UN$SOLD.GET_SOLDT(:datadoc, sTableName, 'ACDE12', pCont=>'217 2165 2114', pDep=>:m_ctdep) создаёт Global Temporary Table, данные которой видны только в той Oracle-сессии, где она была вычислена. Так как каждый вызов Biro26DB — отдельный subprocess/сессия, «запомнить последний расчёт» на уровне Oracle-сессии невозможно между запросами. Поэтому расчёт и сохранение результата выполняются одним execute_script (одна сессия): вызов GET_SOLDT → агрегация по (SC, DEP) → запись в постоянные таблицы YBIRO_STOCK_CALC (шапка прогона) + YBIRO_STOCK_CALC_ITEM (строки, индекс по SC). Последующие запросы читают уже сохранённый результат — быстро, без повторного расчёта. Индекс на самой временной таблице GET_SOLDT не создаётся — Oracle это блокирует сразу после заполнения (ORA-14452); быстрый доступ обеспечивает индекс на YBIRO_STOCK_CALC_ITEM.

На момент внедрения (2026-07-01) в TMDB_CM нет проводок по счетам 217/2165/2114 — реальных остатков ещё нет (Фаза 2 ТЗ, приходы/продажи, не реализована). Поэтому колонка «Кол-во» показывает константу-заглушку (по умолчанию 1000, редактируется прямо в гриде, хранится в localStorage браузера) для товаров без реального остатка — как в исходном Excel. Как только появятся реальные проводки, «Кол-во» будет показывать настоящий остаток.

Деплой: sql/biro26/03_biro26_stock_tables.sql / deploy_biro26_stock_tables.py (11g: seq+trigger).
Эндпоинты: POST /api/biro26/stock/calculate ({data_doc, dep_filter, cont_filter?, pfilt?}), GET /api/biro26/stock/latest, GET /api/biro26/stock/items, GET /api/biro26/products (грид товар+остатки).

Штрихкоды (TMS_MPT_BARCODE) и дерево групп

После импорта штрихкодов (см. IMPORT_TMS_UNIVERS.md §9В/9В.1, import_barcodes.py: 67 235 EAN на 67 080 товаров, до 11 на товар):

  • Поиск по штрихкоду работает в гриде «Marfă / Stoc» и в «Nomenclator» (одно поле поиска: название / артикул / штрихкод; совпадение по любому EAN товара через EXISTS к TMS_MPT_BARCODE).
  • Колонка «Штрихкод» в гриде товаров: первый EAN + бейдж +N, если у товара их несколько.
  • Карточка товара (модалка и детальная панель Nomenclator) показывает все штрихкоды товара чипами.
  • Дерево групп в левой панели «Marfă / Stoc»: GRUPA (12 таможенных групп) → CATEGORIE (~605 товарных групп) с количеством товаров на каждом узле (эндпоинт GET /api/biro26/products/tree, один запрос, ~768 строк). Клик по узлу фильтрует грид (параметры grupa/categorie); «Все товары» и «Сбросить фильтры» снимают выбор. Раскрыта только выбранная группа — дерево остаётся компактным.
  • Попутно: join фида в гриде дедуплицирован (ROW_NUMBER по COD_UNIVERS) — 11 товаров имели по 2 идентичные строки в BIRO26_GOODS и дублировались в выдаче.

Инструкция разработчика по интерфейсу «Marfă / Stoc» (слои, SQL-особенности 11g, состояние/функции фронтенда, как расширять): docs/Biro26/DEV_MARFA_STOC.md.

Редактирование товара и дерева групп («Marfă / Stoc»)

Атрибуты товара редактируются двумя способами: (1) inline в гриде — кнопка ✎ в строке переводит её в режим редактирования (название, группа, категория, ед.изм., цены, бренд; 💾/✖); (2) в карточке товара — кнопка «Редактировать атрибуты» открывает полную форму: поля TMS_UNIVERS (DENUMIREA/NAMERUS/CODVECHI/UM), поля фида BIRO26_GOODS (BRAND/GRUPA/CATEGORIE/ANGRO/IONLINE/RETAIL1), ссылка на картинку (TMS_MPT_TVR.IE_LINKADRES) и штрихкоды (добавить/удалить, TMS_MPT_BARCODE; уникальность EAN контролирует триггер БД — конфликт вернётся ошибкой). Запись — одним атомарным запросом PUT /api/biro26/products/<cod> (поля вне белого списка игнорируются); правки TMS_UNIVERS аудируются штатными history-триггерами ERP.

Дерево групп: у узлов появились значки ✎ (переименовать группу/категорию) и ⇄ (перенести категорию в другую группу) — операции массово правят BIRO26_GOODS.GRUPA/CATEGORIE с подтверждением («будет изменено N позиций»); эндпоинты POST /api/biro26/products/tree/rename|move. Активные фильтры автоматически следуют за переименованием.

Корзина (универсальная)

Кнопка 🛒 в шапке. Товар складывается в корзину с вкладок «Marfă / Stoc», «Stoc (calcul)» и «Nomenclator»: в каждой таблице есть колонка «Заказ» — введённое количество работает как чекбокс, кнопка «В корзину (выбранное)» добавляет все отмеченные позиции разом (несколько позиций за один клик). Повторное добавление того же товара суммирует количество. Корзина хранится в localStorage браузера (biro26_cart) и переживает перезагрузку страницы.

Из корзины (модалка: правка количества, удаление, очистка) данные копируются в буфер обмена:

  • CSV — COD;DENUMIRE;UM;BARCODE;CANT;
  • XML — точный ADO-rowset формат нативного приложения UNA.md (urn:schemas-microsoft-com:rowset, атрибуты NRSET/COD/NRORD/MARK/PARAM/UM/BARCODE); количество передаётся в атрибуте PARAM, NRSET='0', MARK='1', NRORD — порядковый номер.

11.2 Пакет y_ai_BIRO26 (sql/biro26/04_y_ai_biro26.sql)

Создаёт документы «счёт на оплату» (SYSFID 12280, референс COD=140), видимые через нативные представления:

SELECT * FROM VMDB_DOCS_WORK WHERE COD=:COD;
SELECT * FROM VMDB_ST201M WHERE NRDOC=:COD;
SELECT * FROM VMDB_ST201D WHERE NRDOC=:COD ORDER BY RROWID;
  • register_client(p_name) → COD нового клиента TMS_UNIVERS;
  • create_invoice(p_client_cod, p_data) → шапка: TMDB_DOCS (NRSET=MAX+1 по SYSFID, AT2=2 — обходит проверку рабочего периода в TRIG_BFALL_TMDB_DOCS и держит документ видимым в VMDB_DOCS_WORK), проводка через VMDB_ST201M (DT=2214/CT=2171, DTDEP=клиент, CTDEP=1, LEI) и XNRDOC;
  • add_line(p_nrdoc,p_sc,p_cant,p_pret,p_coment) — строка через VMDB_ST201D (SUMA=ROUND(cant*pret,2), RROWID из ID_TMDB_CM);
  • last_doc / get_nrset — COD/NRSET созданного документа (state сессии).

Деплой объектов: ./venv/bin/python deploy_biro26_shop.py (идемпотентен).

11.3 Кнопка «Создать счёт на оплату»

POST /api/biro26/shop/invoice, тело {items:[{cod,qty,name}]}. Два пути: (а) публичный клиент — client_cod из сессии, цены всегда берутся на сервере из BIRO26_GOODS.RETAIL1 (клиентские цены игнорируются); (б) оператор backoffice — передаёт client_cod явно (кнопка в модалке корзины, prompt COD клиента), пустые цены дополняются серверными. 12. Цены по периодам на Marfă / Stoc / period prices on Marfă / Stoc

Цены в гриде Marfă/Stoc (RETAIL, ANGRO, ONLINE) читаются из периодного прайс-листа TPR1D_PERPRLIST (CODPRICE=1 «BIRO») — по тому же принципу, что и вкладка Listă de prețuri — на дату из поля «Prețuri la data» (по умолчанию сегодня, можно менять; параметр price_date у GET /api/biro26/products). Для товаров, которых ещё нет в прайс-листе, fallback — значения из feed-таблицы BIRO26_GOODS. Соответствие колонок: PRETV=RETAIL1, PRETV1=ANGRO, PRETV2=IONLINE.

12.1 Правила периодов (пакет y_ai_BIRO26)

  • set_price(p_sc, p_data, p_pretv, p_pretv1, p_pretv2 [, p_codprice, p_codgrp]) — устанавливает цены, действующие с p_data. Если период с таким DATASTART уже есть — обновление на месте; иначе действующий период ДРОБИТСЯ: нативный INSTEAD OF триггер (TRG_VTPR1D_PERPRLIST_M_ALL на VTPR1D_PERPRLIST) закрывает старый период на p_data-1 и вставляет новый. NULL-параметр цены сохраняет значение колонки из действующего периода.
  • del_price(p_sc, p_data [, p_codprice]) — удаляет период, начинающийся с p_data; соседние периоды ОБЪЕДИНЯЮТСЯ (предыдущий период продлевается до конца удалённого; при удалении первого периода следующий расширяется назад), так что диапазон дат остаётся без разрывов. Когда строка одна — её удалить нельзя (ORA-20261); несуществующий период — ORA-20262.
  • price_on(p_sc, p_data, p_which) — цена на дату ('V' retail / '1' angro / '2' online).

12.2 UI и API

Внизу вкладки Marfă/Stoc — панель «Istoric prețuri»: клик по строке товара в гриде показывает все его периоды (С даты / По дату / цены). У каждой строки кнопка удаления периода (🗑), скрытая, когда строка одна. Изменение цен в inline-редакторе строки и в карточке товара пишется через set_price на дату «Prețuri la data» (дробит период); прочие атрибуты — по-прежнему через PUT /api/biro26/products/<cod>. Endpoints: GET /api/biro26/products/price-history?sc=, POST /api/biro26/products/price {sc, date 'YYYY-MM-DD', retail1, angro, ionline}, POST /api/biro26/products/price/delete {sc, date 'DD.MM.YYYY'}. Цены публичного магазина (shop_prices_for) также читаются из прайс-листа на сегодня.

11А. Универсальный импорт файлов (BIRO26PT) / universal file import

Страница /UNA.md/orasldev/biro26-import-pt (карточка «📥» на лаунчере) — веб-интерфейс поверх PL/SQL-пакета BIRO26PT_importData (вся бизнес-логика в БД; веб только принимает файлы и читает таблицы результата). Поток в 2 фазы:

  1. Загрузка: группа файлов .xlsx/.xls/.csv или один .zip (белый список расширений, защита от path-traversal, лимит 50 МБ, изолированная папка uploads/biro26pt/<uuid>, чистится после обработки) → models/biro26pt_loader.py (subprocess, thick; charset БД CL8MSWIN1251 — кириллица корректна только через python-oracledb) → staging BIRO26PT_RAW/HEADER/FILE;
  2. Dry-run → подтверждение: import_file(p_commit=FALSE) ничего не пишет в прод; UI показывает маппинг колонок (бейджи стратегий HEADER/CONTENT/LAYOUT/MANUAL, ручное переопределение с повторным dry-run — пишется в BIRO26PT_MAP как MANUAL), карточки-счётчики NEW/EXISTING(+цен с расхождением)/AMBIGUOUS/NOARTICOL и превью строк; затем явная кнопка «Importă» → p_commit=TRUE + журнал BIRO26PT_LOG. Импорт идемпотентен (повтор не создаёт дублей).

API (за auth): POST /api/biro26/pt/uploads|analyze|commit|remap, GET /api/biro26/pt/preview/<load_id> (ROWNUM-пагинация — ERP это 11g), GET /api/biro26/pt/help (algo_md из пакета). Спека: /BIRO26/BIRO26PT_WEB_INTERFACE_SPEC.md. Проверено на Set_data_import/3: 407 строк, маппинг по русским заголовкам, повторный commit → +0 товаров/+0 периодов.

11Б. API для десктоп-приложений (una.md/desktop) / machine API

Счёт на оплату и заказ доступны без браузерной сессии — по токену X-API-Key (значение BIRO26_API_TOKEN в .env; пустой = путь отключён; сравнение — constant-time):

GET /api/biro26/shop/report/<invoice|order>/<cod>   → PDF
GET /api/biro26/doc/<cod>                            → JSON (номер, клиент, строки, суммы)
# заголовок: X-API-Key: <token>   (или ?api_key=)

Готовая прослойка: scripts/biro26_report_client.py — только stdlib (на Windows работает со стандартным Python без pip): CLI (biro26_report_client.py 174 [--kind invoice|order|both] [--json] [--out DIR] [--open]), GUI на tkinter (--gui: сервер/токен с сохранением в ~/.biro26_client.ini, COD, чекбоксы форм, скачивание в Downloads и открытие), и класс Biro26Api для импорта в свой код (api.report(cod, kind) → bytes, api.doc(cod) → dict, api.save(cod, kind, dir) → path). Конфиг: аргументы → BIRO26_SERVER/BIRO26_API_TOKEN → ini.

12А. Уведомления о новых счетах/заказах / notifications

Админка: /UNA.md/orasldev/biro26-notify-settings (карточка «🔔» на лаунчере). При создании счёта на оплату (магазин или backoffice) платформа шлёт уведомление «№, клиент, сумма» по включённым каналам — fire-and-forget (фоновый поток; корзина не ждёт и не падает из-за уведомлений):

  • Email — ВСЕ поля настраиваются в админке (host, port, user, пароль, from, SSL, получатели). Секреты при этом в БД не попадают: SMTP-поля админка записывает в .env на сервере (BIRO26_SMTP_*) и применяет сразу без рестарта; пароль обратно не показывается (пустое поле при сохранении = оставить прежний);
  • Telegram — токен бота (@BotFather) и chat_id — в админке;
  • WhatsApp — через бесплатный CallMeBot: добавить их номер в контакты, отправить «I allow callmebot to send me messages», полученный apikey и свой телефон — в админке.

Настройки хранятся в YBIRO_SETTINGS (ключи NOTIFY_*, через y_ai_BIRO26.set_setting). На каждой карточке канала есть кнопка «Trimite test». API (за auth): GET/PUT /api/biro26/notify-settings, POST /api/biro26/notify-test {channel}. Код: models/biro26_notify.py.

13. Варианты товаров (BIRO26_VARIANTS) / product variants

Источник — таблица BIRO26_VARIANTS (см. BIRO26_VARIANTS_IMPLEMENTATION.md): семья = MASTER_COD, у каждого варианта свой COD_UNIVERS, цена — одна на группу (из мастера), одиночки имеют VARIANT IS NULL.

  • Карточка товара (Nomenclator/Marfă): редактируемый блок «Variante (caracteristici)» — все строки семьи, построчное редактирование VARIANT/ARTICOL/FURNIZOR (💾 в строке); изменение VARIANT синхронно обновляет TMS_MPT_BARCODE.COMENT (подпись касс/этикеток); клик по COD открывает карточку этого варианта; текущий товар подсвечен. API: GET /api/biro26/univers/<cod>/variants, PUT /api/biro26/variants/<cod>.
  • Магазин: у товаров с семьёй >1 записи на карточке — селектор характеристики (ленивая загрузка по фокусу, GET /api/biro26/shop/variants?cod=); в корзину и в строку счёта попадает COD_UNIVERS выбранного варианта с полным названием.
  • Каталог (/products, /shop/products) отдаёт variant, master_cod, var_cnt.

14. Печатные формы (jsReport) / printable forms

В ядро платформы добавлен jsReport — отдельный Node.js-сервис (reports/, порт 127.0.0.1:5488, systemd-юнит jsreport.service). Flask отправляет Handlebars-шаблон + данные inline в POST /api/report (recipe chrome-pdf) и получает PDF — store jsReport не используется, шаблоны версионируются в репозитории:

  • reports/templates/biro26_invoice.hbs — «Cont de plată / Счёт-фактура» (двуязычная, реквизиты продавца из BIRO26_FIRM_* в config/.env);
  • reports/templates/biro26_order.hbs — «Comanda cumpărătorului»;
  • reports/templates/helpers.js — формат денег + сумма прописью по-румынски (roAmount).

Корзина: после «Создать счёт на оплату» в магазине появляются кнопки «⬇ Cont de plată (PDF)» и «⬇ Comandă (PDF)»; в backoffice обе формы открываются автоматически. Endpoint: GET /api/biro26/shop/report/<invoice|order>/<cod> — клиент магазина видит только свои документы (DTDEP = его univers_cod), сессия backoffice — любые. Данные берутся из VMDB_ST201M/D + TMS_UNIVERS + YBIRO_CLIENT; НДС считается как включённый (BIRO26_TVA_RATE, по умолчанию 20%).

14.0 Прикрепление PDF к документу (VMDB_DOCS_OLE)

Каждый сгенерированный web-системой счёт/заказ автоматически прикрепляется к документу через нативное представление вложений VMDB_DOCS_OLE (таблица TMDB_DOCS_OLE): NRDOC=COD документа, PFILE=«Cont_de_plata_<COD>.pdf» / «Comanda_<COD>.pdf», OLEOBJ=PDF (BLOB), комментарий «... (web/PDF)». Одно вложение на (документ, тип формы): повторная генерация заменяет файл, а не дублирует. Прикрепление — best effort: если оно не удалось, скачивание PDF всё равно работает. Проверка: SELECT * FROM VMDB_DOCS_OLE WHERE NRDOC = :COD. Технически BLOB передаётся воркеру через JSON-параметр {"__b64__": ...} с явным биндом DB_TYPE_BLOB (иначе >32KB уходит как LONG → ORA-01461).

14.1 Два движка: jsReport и pdfme (визуальный редактор)

Каждая форма (счёт/заказ) может рендериться на выбор одним из двух движков; активный движок хранится в reports/templates/engines.json и переключается в админке шаблонов (блок «Motor activ»):

  • jsReport — Handlebars + chrome-pdf (пиксельная верстка HTML/CSS, шаблоны biro26_*.hbs);
  • pdfme (pdfme.com, MIT) — шаблон-JSON + pdf-lib (без Chromium, рендер быстрее и легче; шаблоны pdfme_invoice.json / pdfme_order.json). Редактируются визуально drag&drop в Designer: /UNA.md/orasldev/biro26-pdfme-designer (кнопка «🎨 pdfme Designer» в админке; библиотека Designer грузится с CDN только в браузер оператора). Сумма прописью для pdfme считается на стороне Flask (_ro_amount); итоги добавляются строками таблицы, поэтому длинные списки не наезжают на блок итогов. Шрифт DejaVu Sans (кириллица+диакритика) — reports/fonts/ и static/biro26/fonts/. Endpoint генерации в сайдкаре: POST /pdfme/generate {template, inputs}.

Редактор шаблонов (админка): /UNA.md/orasldev/biro26-report-templates (ссылка также на лаунчере модуля) — список файлов reports/templates/, правка в браузере, предпросмотр PDF несохранённого текста на демо-данных или по COD реального документа, сохранение с бэкапом <имя>.bak. API (за auth): GET/PUT /api/biro26/report-templates[/<name>], POST /api/biro26/report-templates/preview. Правки на сервере живут до следующего деплоя кода — переносите их и в репозиторий.

Запуск локально: cd reports && npm install && node server.js (Studio: http://127.0.0.1:5488). Сервер: Node 20 (NodeSource), зависимости Chromium через apt, юнит с MemoryHigh=450M/MemoryMax=600M (на машине <1 GB RAM; 2 GB swap). Первый рендер ~20 с (холодный старт Chromium), дальше ~5–7 с. Если сервис лежит, API отвечает report service unavailable — счёт при этом создаётся как обычно.
